# 무료 진단 어뷰징 방지 정책

작성일: 2026-07-02 · 적용 대상: 무료 진단 신청폼 및 접수 서버

무료 배포 모델은 비용(API 호출·리포트 생성·상담 리소스)이 우리 쪽에 있으므로,
동일인·동일 IP의 반복 신청과 봇 제출을 막는 것이 운영 안정성의 핵심이다.
IP·User-Agent는 개인정보처리방침의 "자동 수집 항목"으로 이미 고지되어 있다.

## 1. 요청 제한 (기본값 — 운영하며 조정)

| 기준 | 제한 | 초과 시 |
|---|---|---|
| IP당 | 일 3회 / 월 10회 | 자동 반려 + 관리자 검토 큐 |
| 이메일당 | 월 2회 | 자동 반려 |
| 전화번호당 | 월 3회 | 자동 반려 |
| 병원(병원명+주소)당 | 월 1회 | 안내 후 반려 — "동일 병원은 월 1회 진단" (갱신은 익월) |
| 차단 승격 | 임계 2배 초과 또는 반려 3회 누적 | `blocked` — 관리자 해제 전 재신청 불가 |

반려 안내 문구(신청자에게 노출)는 사유를 뭉뚱그린다: "요청이 많아 접수가 제한되었습니다.
영업일 기준 1일 후 다시 시도하거나 문의처로 연락해 주세요." (임계값 노출 = 우회 힌트가 되므로 금지)

## 2. 탐지 신호 (관리자단 어뷰징 모니터에 표시)

- 같은 IP에서 서로 다른 병원명으로 다수 신청 (경쟁 정보 수집 의심 — 대행사/브로커)
- 같은 신청자(이메일/전화)가 서로 다른 병원 다수 신청
- 폼 체류 시간 3초 미만 제출, 허니팟 필드 입력 (봇)
- 일회용 이메일 도메인, 존재하지 않는 병원명 반복
- 야간 대량 제출, User-Agent 없음/자동화 도구 표기

## 3. 방어 계층

| 계층 | 수단 | 상태 |
|---|---|---|
| 클라이언트 (우회 가능 — UX용) | 허니팟 필드, 최소 체류시간 검사, localStorage 일 2회 안내 | ✅ 신청폼에 구현 |
| 접수 서버 (실효 방어) | 위 1장 제한을 `diagnosis_requests` 로그 기준으로 검사 (`count_recent_requests`), `abuse_blocklist` 대조 | 스키마 준비 완료 — 서버 연동 시 활성 |
| 인프라 | 호스팅/프록시 레벨 rate limit(예: Cloudflare), 필요 시 CAPTCHA(트래픽 이상 시에만) | 서버 결정 후 |
| 운영 | 관리자단 어뷰징 모니터에서 검토·차단/해제, 조치 기록 | ✅ 관리자단 프로토타입 |

주의: 클라이언트 검사(localStorage 등)는 시크릿 창으로 쉽게 우회된다.
**실제 차단은 반드시 서버에서** 한다. 클라이언트 가드는 선량한 사용자의 실수 방지용이다.

## 4. 개인정보 관점

- IP·UA 보관 목적: 부정 이용 방지·분쟁 대응 (처리방침 1조 5항, 2조 자동수집 항목에 고지됨)
- 보관 기간: 신청 데이터와 동일 (리포트 제공 후 1년), 차단 목록은 차단 해제 시까지
- IP만으로 특정 개인을 식별하지 않으며, 차단 판단 외 목적으로 사용하지 않는다
- 수신거부(suppression_list)는 재전송 방지 목적상 3년 보관 (처리방침 3조 3항)
