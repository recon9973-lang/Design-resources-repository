# 베놈 공통 자격증명(토큰) 표준 — 프로젝트 간 통일 규격 (2026-07-04)

목적: 여러 프로젝트/저장소가 **같은 API 키(토큰)를 같은 방식으로** 쓰도록 통일한다.
핵심 원칙 3가지만 지키면 프로젝트가 몇 개든 자동으로 통일된다.

## 원칙 1 — 키는 계정당 하나, 전 프로젝트가 공유

네이버 검색광고 라이선스·NCP Maps 키·SGIS 키·공공데이터 키는 **계정 단위**다.
프로젝트마다 새로 발급하지 말고 **하나를 발급해 모든 프로젝트가 공유**한다.
(프로젝트별로 키를 따로 파면 한도·정산·회전 관리가 파편화된다.)

## 원칙 2 — 환경변수 이름을 전 프로젝트 동일하게 고정

어느 저장소·언어든 아래 **정확한 이름**만 쓴다. 이름이 같으면 같은 값이 흘러든다.

| 용도 | 환경변수 | 발급처 |
|---|---|---|
| 네이버 오픈 API(검색·지역) | `NAVER_CLIENT_ID` / `NAVER_CLIENT_SECRET` | developers.naver.com |
| **네이버 검색광고(검색량·연관검색어)** | `NAVER_AD_API_KEY` / `NAVER_AD_SECRET_KEY` / `NAVER_AD_CUSTOMER_ID` | searchad.naver.com |
| 네이버 클라우드 Maps(정적 지도) | `NAVER_MAP_KEY_ID` / `NAVER_MAP_KEY` | console.ncloud.com |
| SGIS 통계지리 | `SGIS_CONSUMER_KEY` / `SGIS_CONSUMER_SECRET` | sgis.kostat.go.kr |
| 공공데이터(HIRA 등) | `DATA_GO_KR_SERVICE_KEY` | data.go.kr |

> 다른 프로젝트가 `SEARCHAD_KEY`, `NAVER_AD_LICENSE` 등 **다른 이름**을 쓰고 있으면
> 위 표준 이름으로 **리네이밍**하는 것이 "통일" 작업의 실체다.

## 원칙 3 — 코드는 단일 지점(config)에서만 키를 읽는다

각 파일이 `os.environ`/`process.env`를 직접 읽지 말고, **config 접근자 함수 한 곳**만
호출한다. 키 이름 변경·회전 시 그 한 곳만 고치면 된다.
(hospital-marketing 파이프라인은 `medirank/config.py`의 `searchad_creds()` 등으로 이미 적용됨.)

**Python 참고 구현** (그대로 복사해 쓰는 표준 패턴):
```python
# config.py — 프로젝트마다 이 파일 하나만 두고, 나머지는 여기서만 읽는다
import os

def searchad_creds() -> tuple[str, str, str]:
    return (os.environ.get("NAVER_AD_API_KEY", ""),
            os.environ.get("NAVER_AD_SECRET_KEY", ""),
            os.environ.get("NAVER_AD_CUSTOMER_ID", ""))

def searchad_available() -> bool:
    return all(searchad_creds())
```

**Node 참고 구현**:
```js
// config.js
const searchadCreds = () => [
  process.env.NAVER_AD_API_KEY || "",
  process.env.NAVER_AD_SECRET_KEY || "",
  process.env.NAVER_AD_CUSTOMER_ID || "",
];
const searchadAvailable = () => searchadCreds().every(Boolean);
module.exports = { searchadCreds, searchadAvailable };
```

## 여러 저장소가 한 값을 공유하는 방법

### A. GitHub Organization Secrets (추천 — repo가 여럼일 때)
1. GitHub 조직 → Settings → Secrets and variables → Actions → **Organization secrets**
2. 위 표준 이름으로 시크릿 등록(예: `NAVER_AD_API_KEY`), 접근 대상 repo 선택
3. 각 repo의 워크플로에서 `${{ secrets.NAVER_AD_API_KEY }}`로 주입 → **값은 한 곳에서만 관리**
4. 키 회전 시 조직 시크릿 한 번만 갱신하면 전 repo 반영

### B. 공용 `.env` 배포 (서버/PC 로컬 실행 시)
- 표준 이름으로 채운 `.env` 하나를 각 프로젝트 루트에 배치(또는 심볼릭 링크).
- `.env`는 절대 커밋 금지(각 repo `.gitignore`에 포함) — 값은 안전한 채널로만 공유.

## 새 프로젝트 온보딩 체크리스트

- [ ] 키 값은 새로 발급하지 말고 기존 공유 키 재사용
- [ ] `.env` gitignore 확인, 커밋 이력에 키 유출 없는지 점검
