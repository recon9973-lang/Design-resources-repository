# 원장 앱 리드 접수 — Supabase 구축 가이드 (2026-07-04)

죽어 있던 landing 신청 폼을 **실제 저장 + 관리자 조회**로 연결한다. 단일 Postgres
(Supabase)에 리드를 동의 증적과 함께 저장하고, 서버에서 어뷰징을 검사한다.

## 0. 준비물
- Supabase 프로젝트 1개(무료 티어 가능) — 진단 서비스 전용(본사이트 DB와 분리)
- Supabase CLI (`npm i -g supabase` 또는 brew)

## 1. 스키마 적용 (SQL Editor 또는 CLI)
순서대로 실행:
```
db/schema.sql              # 관리자단 기본 스키마(병원·리드·어뷰징 등)
db/schema_member_app.sql   # 원장 앱 델타(인증·자가진단·리포트 게이팅)
db/policies_member_app.sql # RLS 기준선(deny-by-default + 소유자 정책)
```
> PostGIS·CITEXT·pgcrypto 확장은 스크립트가 `CREATE EXTENSION`으로 켠다.
> Supabase는 세 확장 모두 기본 지원.

## 2. 리드 접수 Edge Function 배포
```
supabase link --project-ref <프로젝트-ref>
supabase functions deploy leads --no-verify-jwt   # 공개 폼이라 JWT 미검증
```
- `SUPABASE_URL` / `SUPABASE_SERVICE_ROLE_KEY` 는 런타임에 자동 주입됨.
- (권장) 허용 오리진 좁히기:
  ```
  supabase secrets set ALLOWED_ORIGIN=https://diagnosis.베놈도메인
  ```
  미설정 시 `*`(모든 오리진 허용) — 배포 후 반드시 좁힐 것.

배포 URL: `https://<프로젝트-ref>.supabase.co/functions/v1/leads`

## 3. 프론트 연결
`landing/index.html` 상단의 상수를 배포 URL로 교체:
```js
const LEADS_ENDPOINT = 'https://<프로젝트-ref>.supabase.co/functions/v1/leads';
```
- 비워 두면 기존처럼 **프로토타입(무저장)** 으로 동작한다(안전 폴백).
- CSP `connect-src`에 `https://*.supabase.co` 는 이미 추가돼 있다.

## 4. 동작 확인
```
curl -sX POST "$LEADS_ENDPOINT" -H 'content-type: application/json' -d '{
  "hospital":"테스트의원","department":"피부과","address":"서울 강남구 테헤란로 1",
  "name":"홍길동","role":"원장","phone":"010-1234-5678","email":"t@example.com",
  "delivery":"이메일","consent_required":true }'
# → {"ok":true,"id":1}
```
- SQL Editor에서 `SELECT id,hospital_name,email,status,created_at FROM diagnosis_requests ORDER BY id DESC;`
- 동의 증적(`consent_text_version`,`client_ip`,`user_agent`)이 함께 저장됐는지 확인.

## 5. 서버가 막는 것 (클라이언트 우회 불가)
| 상황 | 응답 | reason |
|---|---|---|
| 허니팟 채워짐(봇) | 200(무저장) | skipped |
| 필수값/이메일/필수동의 누락 | 400 | missing_fields·bad_email·consent_required |
| 차단 목록(IP·이메일·전화) | 403 | blocked |
| IP 일3/월10·이메일 월2·전화 월3 초과 | 429 | rate_limited |
| 동일 병원 월 1회 초과 | 429 | duplicate_hospital |

프론트의 3초 트랩·일일 한도(localStorage)는 **UX 사전 필터**일 뿐, 진짜 판정은 서버가 한다.

## 5-b. 실동작 검증 결과 (2026-07-04, 로컬 실 DB)
클라우드 없이 **로컬 PostgreSQL 16**에 `schema.sql`+`schema_member_app.sql`을 적용하고
접수 로직을 실제로 돌린 결과(스키마·SQL·어뷰징 규칙은 운영과 동일):

| # | 시나리오 | 응답 | 저장 |
|---|---|---|---|
| ① | 정상 신청(동의 완비) | `{ok:true,id:1}` | 저장 + 동의 증적(`consent_text_version`,`client_ip`,채널동의) |
| ② | 허니팟 채워짐 | `{ok:true,skipped:true}` | 무저장 |
| ③ | 필수 동의 누락 | `400 consent_required` | 무저장 |
| ④ | 같은 병원 재신청 | `429 duplicate_hospital` | 무저장 |
| ⑤ | 동일 이메일 3회(한도 2) | 1·2 저장, 3번째 `429 rate_limited` | 2건만 |

> 참고: 이 샌드박스엔 Docker/Supabase CLI/Deno가 없어 접수 로직을 **동일 규칙의
> Node 포팅**으로 실 DB에 붙여 검증했다. 운영은 `supabase/functions/leads/index.ts`
> (Deno) 그대로 배포하며 스키마·쿼리·판정은 검증본과 동일하다. 또한 `schema.sql`은
> PostGIS 함수(`ST_DWithin`)를 쓰는 반경 질의만 PostGIS를 요구하며(리드 접수와 무관),
> Supabase는 PostGIS를 기본 지원한다.
