# 원장 앱 DB 필요성 검토 & 아키텍처 설계 (2026-07-04)

질문: **원장(고객)-facing 앱도 자체 DB가 필요한가?** 필요하면 아키텍처를 구성하고 설계를 시작한다.

## 1. 현재 상태 (근거)

| 구성 | 현재 | persistence |
|---|---|---|
| 자가진단 계산기(리스크·비용·동선) | 순수 client-side, PII 없음, 서버 호출 0 | **불필요** |
| 마킹 리포트 미리보기 | auto_diagnose가 정적 HTML(마킹판/전체) 생성, GitHub Pages | 게이팅 없음 |
| 신청 폼(`landing/applyForm`) | 필드·허니팟 완비, 그러나 **백엔드 미연결**(`form-action 'self'`, 서버 없음) → 제출해도 저장 안 됨 | **비어 있음(급함)** |
| 관리자단 스키마(`db/schema.sql`) | Postgres+PostGIS. `users`(role `customer` 포함)·`diagnosis_requests`(리드+동의증적)·`suppression_list`·`abuse_blocklist` **이미 설계됨** | 설계만, **런타임 없음** |

즉 데이터 모델은 상당 부분 존재하나, **이 스키마를 돌리는 서버가 아직 없다**(정적만 배포). 신청 폼은 사실상 죽어 있다.

## 2. 결론

### 2.1 persistence는 필요하다 — 기능이 요구한다
| 기능 | DB 필요 | 이유 |
|---|:--:|---|
| 자가진단 계산 | ✗ | 클라이언트 계산, 유지 |
| 마킹 미리보기 | ✗ | 정적 |
| **무료 회원가입 → 전체 공개** | ✓ | 계정+열람권한 저장, 정적 불가 |
| **상담/리드 신청** | ✓ | 제출 저장 + 차트룸 인입 |
| **관리자 차트룸 실시간 인입** | ✓ | 공유 store에서 조회 |
| **재방문 시 저장 결과 재열람** | ✓ | 회원 키에 저장 |
| **어뷰징 방지(재신청 차단)** | ✓ | 서버 상태여야 함(클라 우회 불가) |

### 2.2 그러나 '자체(별도) DB'는 불필요 → 관리자 차트룸과 **단일 DB 공유**
근거:
- **리드 생애주기가 하나다**: 원장 신청 → 관리자 상담 → 광고대행 전환. 같은 행(`diagnosis_requests`)을 양쪽이 본다. 별도 DB면 동기화 부담 + 정합성 위험.
- **PII 거버넌스**: 민감정보(의료 인접·개인정보)를 두 곳에 두면 동의 증적·파기·감사가 이중화. 단일 store + `audit_logs`가 법무상 안전.
- **스키마 재사용**: 이미 customer/lead 테이블 포함.

→ 원장 앱 = **「정적 프론트 + 인증 + owner-스코프 API」를 기존 단일 Postgres 위에** 얹는다. **자체 DB는 만들지 않는다.**

> 뉘앙스: "원장 앱 전용 별도 DB"가 아니라 **"진단 서비스(원장앱+차트룸) 공용 DB 1개"**. 베놈 본사이트가 워드프레스 등 다른 DB를 쓰더라도, 진단 서비스는 그와 **분리된 전용 DB 1개**를 갖는 게 맞다(민감정보·스키마 부적합 이유로 본사이트 DB에 섞지 말 것).

## 3. 아키텍처

```
[서브도메인 diagnosis.베놈도메인 — 정적 프론트 (GitHub Pages/CDN)]
   ├ 자가진단 계산기            (client-only, 서버 X)
   ├ 마킹 리포트 뷰어           (권한 따라 masked/full)
   └ 회원가입·로그인·신청 폼 ──fetch──┐
                                      ▼
                        [ /api  인증·접수 서버 ]
                        - 리드 접수(동의 증적·어뷰징 카운터)
                        - magic-link 회원가입/세션
                        - 자가진단 결과 저장·리포트 열람 grant
                                      │
                        [ 단일 PostgreSQL (+PostGIS) ]   ← db/schema.sql + 델타
                                      ▲
        [관리자 차트룸 admin/]  ──── 같은 DB 조회(role 분리·RLS)
        [파이프라인 auto_diagnose] ── reports 등록 → 게이팅 열람
```

### 3.1 스택 추천 — **Supabase** (Postgres + Auth + RLS + Edge Functions + Storage)
- 기존 **Postgres 스키마를 그대로 이식**, magic-link/OTP 회원가입 기본 제공, **RLS로 owner-스코프**(원장은 본인 데이터만), Edge Function으로 어뷰징 카운터·리드 접수, Storage로 리포트 HTML/PDF 보관. 운영부담 낮고 무료 티어로 파일럿 가능.
- **대안 A**: 자체 FastAPI + Postgres(Neon/RDS) — 통제권↑, 운영부담↑. 스키마·RLS 개념 동일.
- **대안 B(본사이트가 WordPress면)**: WP는 콘텐츠만, 진단 API는 서브도메인에 분리. **WP DB에 리드/회원 넣지 말 것.**
- **비권장**: 구글폼/Formspree 등 정적 폼만 — 동의 증적·어뷰징·회원·차트룸 연동 불가, 법무 리스크.

### 3.2 보안·개인정보 (민감·의료 인접)
- **최소수집**: 자가진단 계산값은 서버 전송 안 함 — 회원이 "저장"을 눌러야만 저장.
- **동의 증적**: `diagnosis_requests`에 이미 일시/IP/문구버전 보관. 회원가입에도 동일 적용.
- **RLS**: `customer`는 own rows만. `data_operator`/`legal_reviewer`/`admin` role 분리(스키마 `user_role` 재사용).
- **audit_logs**: 열람·다운로드·삭제 기록(스키마 존재).
- **파기**: 리텐션 잡(신청 N개월 후 자동 파기). HTTPS·서브도메인 격리·토큰 만료.

## 4. 설계 시작 — 스키마 델타 & API

### 4.1 기존 스키마에 없는 것 (원장 앱 전용, **additive**) → `db/schema_member_app.sql`
- `auth_challenges` — magic-link/OTP 발급·검증(무비밀번호 회원가입)
- `self_check_results` — 자가진단(리스크·비용·동선) 결과 저장/재열람
- `reports` — 생성 리포트 레지스트리(masked/full 경로·공개 토큰)
- `report_grants` — 회원 → 리포트 전체공개 열람 해제

기존 `db/schema.sql`은 **건드리지 않고** 확장한다(리드/회원/어뷰징은 그대로 재사용).

### 4.2 API 엔드포인트 초안 (원장 앱)
| 메서드·경로 | 용도 | 인증 |
|---|---|---|
| `POST /api/leads` | 신청 접수(어뷰징 검사·동의 증적) → `diagnosis_requests` | 불필요 |
| `POST /api/auth/start` | 이메일 → magic-link/OTP 발송 | 불필요 |
| `POST /api/auth/verify` | 토큰 → 세션 발급 | 불필요 |
| `POST /api/self-check` | 자가진단 결과 저장 | 회원 |
| `GET /api/me/self-checks` | 내 저장 결과 목록 | 회원 |
| `POST /api/reports/{token}/unlock` | 회원가입 후 전체공개 grant | 회원 |
| `GET /api/reports/{token}` | 권한 따라 masked/full 반환 | 선택 |

## 5. 단계별 로드맵
1. **(지금)** 검토·아키텍처·스키마 델타·API 초안 ← 본 문서 + `schema_member_app.sql`
2. 스택 확정(Supabase 권장) → 프로젝트 생성 → `schema.sql` + 델타 적용
3. **`POST /api/leads` + 어뷰징 + landing 폼 연결** — *가장 급함*(지금 폼이 죽어 있음)
4. 회원가입(magic-link) + 마킹 해제 열람
5. 자가진단 저장/재열람
6. 관리자 차트룸 실데이터 연결
